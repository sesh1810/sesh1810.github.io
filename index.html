Diagnosing a Cascading Failure in Instagram Reels Ranking

How a routine model update triggered a 4-hour content quality degradation across 340M+ daily active users

Context

Instagram Reels serves billions of short-form video recommendations daily through a multi-stage ranking pipeline. The system scores candidate content for relevance, filters it through safety classifiers, and delivers personalized feeds to users across 190+ countries. As Technical Program Manager owning the ranking platform reliability roadmap, this post-mortem documents how a subtle regression cascaded through tightly coupled services and what we built to prevent recurrence.

Problem Statement

On a routine Tuesday deployment, Ranking Model v2.7.1 rolled out to production. The update introduced an improved engagement predictor that had passed all offline evaluation benchmarks. Within two hours of full rollout, user-facing content quality degraded significantly — users reported repetitive, low-relevance, and stale content. The incident lasted 4 hours, affecting an estimated 340M daily active users before the model was rolled back.

The failure was not in the model itself. It was in how the model's output distribution interacted with a downstream system that nobody expected to be affected.

Root Cause Analysis: 5 Whys

  1. Why did content quality degrade? → Safety classifiers timed out, triggering a fallback to stale cached rankings (some up to 6 hours old)
  2. Why did safety classifiers time out? → They received 3.2x their normal volume of "borderline" content requiring full classification passes
  3. Why did borderline volume spike? → The ranking model's output score distribution shifted, compressing more candidates into the 0.45–0.65 safety threshold band
  4. Why did scores shift? → Model v2.7.1 used updated feature normalization statistics (retrained population means) without recalibrating the downstream threshold boundaries
  5. Why wasn't this caught in testing? → Canary evaluation monitored per-request performance (latency, errors) but not output distribution shape. The model produced valid scores — they were just distributed differently
Core Insight

The failure wasn't in any single component. Every service behaved correctly in isolation. The cascade emerged from an implicit contract between ranking and safety: the safety system was provisioned for the historical score distribution, not the one v2.7.1 produced. No alert fired because no individual metric crossed its threshold.

Remediation Plan

I drove the following action items across Ranking, Safety, and Infrastructure teams with a 30-day completion target:

  1. Score Distribution Monitoring — Added KL-divergence tracking between canary and baseline score distributions. Canary auto-halts if divergence exceeds 0.15 for 10 consecutive minutes
  2. Circuit Breaker — Implemented a circuit breaker between ranking and safety services. When safety queue depth exceeds 2x baseline, the system degrades gracefully by tightening the ranking cutoff rather than flooding safety
  3. Cache Freshness SLO — Established a 30-minute maximum staleness policy for fallback caches. Cache entries older than the SLO are discarded rather than served, triggering a degraded-but-fresh ranking path
  4. Cross-Service Contract Testing — Introduced integration tests that validate output distribution assumptions between coupled services before any model deployment

Key Takeaways

  • Monitor contracts, not just components. Per-service health metrics missed a failure that lived in the interaction between services
  • Fallback systems need SLOs too. A stale cache served "successfully" for hours — but success without freshness is a silent failure
  • Distribution shifts are invisible to threshold alerts. Every individual score was valid. The problem was the shape of the aggregate distribution
Wayve Migration Runbook 9 min read

Zero-Downtime Migration of Safety-Critical Perception Pipelines

Moving autonomous driving inference from a monolithic GPU cluster to cloud-native microservices without compromising safety

Context

Wayve's perception system is the sensory cortex of its autonomous vehicles: it processes raw camera, LiDAR, and radar inputs into real-time object detections, lane boundaries, and depth maps that feed directly into the motion planning stack. As the fleet scales from 25 to 300+ test vehicles across London and other cities, the current monolithic architecture cannot scale. As Technical Program Manager leading this migration, the challenge is not just technical — it's coordinating across perception engineering, safety validation, fleet operations, and regulatory compliance to make a zero-downtime transition on a safety-critical system.

Problem Statement

The current perception pipeline runs as a single monolithic process on a custom GPU cluster. Three tightly-coupled inference stages — object detection (YOLOv8-derived), lane segmentation (custom U-Net), and monocular depth estimation — share GPU memory and execute sequentially in a single process. This creates three critical scaling bottlenecks:

  • No independent scaling: A 2x improvement to object detection requires redeploying the entire pipeline, including unchanged models
  • Single point of failure: A crash in depth estimation kills all three inference stages simultaneously
  • Fleet scaling wall: The cluster maxes out at ~40 concurrent vehicle streams. Wayve's roadmap requires 300+ by EOY

The migration must maintain the hard safety constraint of <50ms end-to-end inference latency (sensor input to planning output), certified to ASIL-B functional safety standards.

Migration Phases

Phase 1
Wk 1–3
Instrumentation & Baseline. Instrument the monolithic pipeline with per-stage latency tracing, memory profiling, and output checksums. Establish quantitative baselines: p50/p99 latency per stage, throughput ceiling, and failure modes. These baselines become the acceptance criteria for the new architecture.
Phase 2
Wk 4–8
Parallel Architecture Build. Deploy the three inference stages as independent Kubernetes services with gRPC interfaces. Introduce a Fusion Service that aggregates outputs and presents a unified interface to Planning. Critical: implement a latency budget allocator that dynamically distributes the 50ms budget across services based on current load.
Phase 3
Wk 9–14
Shadow Traffic & Safety Validation. Run both architectures simultaneously. The new pipeline processes real sensor data in shadow mode, with outputs compared bit-for-bit against the monolith. Safety team runs the full ASIL-B validation suite against the new architecture. Acceptance gate: <0.01% output divergence across 10M inference cycles and zero safety-critical mismatches.
Phase 4
Wk 15–18
Progressive Cutover. Migrate vehicles in cohorts of 25, starting with the depot test fleet (non-public-road). Each cohort runs on the new pipeline for 72 hours with enhanced monitoring before the next cohort migrates. Rollback is instant: a config flag switches any vehicle back to monolithic processing within one inference cycle.

Stakeholder Coordination

The hardest part of this migration isn't the Kubernetes manifest. It's that the Perception team, Safety team, Fleet Ops, and Regulatory each have veto power, different risk tolerances, and competing timelines. The TPM's job is to create a shared definition of "safe enough to proceed" before Phase 3 begins, not during it.

Key Takeaways

  • Safety-critical migrations require parallel validation, not sequential. You cannot test-then-cutover; you must run both systems simultaneously long enough to build statistical confidence
  • The rollback plan IS the migration plan. Every phase must be instantly reversible. If rollback is hard, the migration architecture is wrong
  • Define acceptance criteria before building. Phase 1's instrumentation work prevents Phase 3 from becoming an open-ended debate about what "good enough" means

Deprecating Legacy Infrastructure Without Breaking the Internet

An 18-month lifecycle plan for sunsetting Stripe's original payment gateway while protecting 12,000 merchant integrations

Context

Stripe's original payment processing gateway — internally called "Gateway Classic" — was built during the company's first year. A decade later, it still processes 8% of total transaction volume from approximately 12,000 long-tail merchants. As Technical Program Manager leading the deprecation initiative, the challenge is that these merchants depend on undocumented API behaviors, legacy webhook formats, and error codes that don't exist in the modern gateway. A forced cutover would break payment processing for small businesses that depend on Stripe as their sole payment provider.

Problem Statement

Gateway Classic presents three escalating risks:

  • Security: Uses TLS 1.1, approaching end-of-support. PCI DSS 4.0 requires TLS 1.2+ by March next year. Non-compliance exposes Stripe to audit findings and merchant liability
  • Operational cost: Requires 3.2x the engineering maintenance hours per transaction vs. the modern gateway. Two dedicated engineers spend ~60% of their time on Classic-specific incident response
  • Scaling ceiling: No horizontal scaling capability. During Black Friday, Classic hits its throughput limit and begins dropping transactions for its merchants

The constraint: zero payment processing interruptions. Even one dropped transaction during migration means lost revenue for a small business owner. And unlike a B2B SaaS migration, we cannot schedule "maintenance windows" — payments happen 24/7.

The Compatibility Shim is the key architectural decision. Rather than forcing merchants to rewrite their integrations, we build a translation layer that accepts Classic API calls and converts them to modern gateway requests. This lets us decommission Classic's infrastructure without requiring any merchant code changes for the initial cutover.

Phased Deprecation Plan

Phase 1
Month 1–3
Audit & Shim Development. Catalog every undocumented API behavior, legacy error code, and custom webhook format used by Classic merchants. Build the Compatibility Shim with 100% behavioral parity. Instrument Classic to capture full request/response pairs for replay testing.
Phase 2
Month 4–6
Shadow Traffic Validation. Mirror 100% of Classic traffic through the Shim → Modern Gateway path. Compare responses. Acceptance gate: <0.001% response divergence across 30 days (excluding expected timing differences). Fix every divergence — no exceptions.
Phase 3
Month 7–12
Tiered Migration. Three merchant tiers based on integration complexity:
• Self-serve (8,200 merchants): Simple integrations. Auto-migrate through Shim with email notification. Provide one-click rollback button in dashboard
• Assisted (3,400 merchants): Medium complexity. Dedicated migration guide, sandbox environment, and support ticket priority queue
• White-glove (400 merchants): High complexity or high revenue. Assigned migration engineer, custom testing, direct Slack channel for live support
Phase 4
Month 13–18
Shim Sunset & Direct Migration. Incentivize merchants to adopt the modern API directly (shim adds ~8ms latency). Provide automated code migration tools, updated SDKs, and migration credits. Target: 80%+ merchants migrated off shim to direct modern API. Decommission Classic infrastructure. Shim remains for long-tail.

Communication Strategy

Deprecation is as much a communication challenge as a technical one. The plan follows a structured cadence:

  • T-12 months: Public blog post announcing deprecation timeline. No merchant action required yet
  • T-9 months: Personalized email to each merchant with their specific migration tier and timeline. Link to migration guide
  • T-6 months: Dashboard banner in Stripe console. Sandbox environment available for testing
  • T-3 months: Direct outreach to merchants who haven't begun migration. Offer migration support calls
  • T-1 month: Final notice with specific cutover date. Escalation path for merchants needing extensions
The Long Tail Problem

The last 2% of merchants will take 50% of the effort. These are merchants with custom integrations built by contractors who are no longer available, merchants in industries with regulatory constraints on software changes, or merchants who simply don't read emails. The white-glove tier exists specifically for this long tail. Budget for it from day one.

Key Takeaways

  • Build the bridge before closing the road. The Compatibility Shim lets us decommission legacy infrastructure without requiring merchant action. Migration becomes optional (but incentivized), not forced
  • Segment merchants by integration complexity, not revenue. A $500/month merchant with a deeply custom integration needs more support than a $50,000/month merchant using the standard SDK
  • Deprecation timelines are communication artifacts, not engineering artifacts. The technical migration can happen in 6 months. The 18-month timeline exists to give merchants time to plan, budget, and execute their side of the change
